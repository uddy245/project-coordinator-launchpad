# FND-003: Environment variable plumbing with `@t3-oss/env-nextjs`

**Milestone:** M1 Foundation
**Dependencies:** FND-001
**Estimated session length:** short (≤30 min)
**Status:** not started

## Context

Typed, validated env access so every downstream ticket can rely on `env.ANTHROPIC_API_KEY` etc. without `process.env` guessing. `src/env.ts` is the single import point. Missing required vars should fail the build with a readable error, not mysteriously at runtime.

## Questions to resolve before starting

- None. The `@t3-oss/env-nextjs` pattern is well-documented at https://env.t3.gg/docs/nextjs.

## Files to create

- `src/env.ts` — Zod-validated env definition with `server` and `client` splits
- Update `.env.example` with every variable listed in BUILD_PLAN.md Appendix I

## Acceptance criteria

- [ ] `src/env.ts` exports a typed `env` object
- [ ] Missing required env var fails `pnpm build` with a readable Zod error
- [ ] Public vars (`NEXT_PUBLIC_*`) are accessible in client components; server-only vars are not
- [ ] Import pattern is `import { env } from "@/env"` everywhere (add to CLAUDE.md rules)
- [ ] `ANTHROPIC_SPEND_CAP_USD` defaults to 100 when unset

## Tests required

- [ ] Unit: `tests/unit/env.test.ts` — mock `process.env`, assert validation throws on missing required, parses on valid
