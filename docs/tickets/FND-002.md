# FND-002: Wire up Supabase (local + remote) with `@supabase/ssr`

**Milestone:** M1 Foundation
**Dependencies:** FND-001
**Estimated session length:** medium (30–90 min)
**Status:** not started

## Context

Add Supabase with the current `@supabase/ssr` package, per [Supabase's official Next.js App Router SSR guide](https://supabase.com/docs/guides/auth/server-side/nextjs). Three client helpers (browser, server, middleware), middleware that refreshes the session on every request, and a working local Supabase instance via the CLI.

This ticket does not implement auth UI (that's AUTH-001). It only wires up the plumbing so future tickets can create clients and run queries.

## Questions to resolve before starting

- None. The Supabase SSR pattern is documented in the guide linked above. Follow it verbatim, including the cookie `getAll`/`setAll` shape.

## Files to create

**Create in `src/lib/supabase/`:**
- `client.ts` — browser client using `createBrowserClient`
- `server.ts` — server client using `createServerClient` reading from `next/headers` `cookies()`
- `middleware.ts` — `updateSession(request)` that refreshes auth token and returns `NextResponse`

**Create at repo root:**
- `src/middleware.ts` — Next.js middleware calling `updateSession`, with matcher excluding static assets
- `.env.example` (update with Supabase vars — see BUILD_PLAN.md Appendix I)

**Create via `supabase init`:**
- `supabase/config.toml`
- `.gitignore` additions: `supabase/.branches`, `supabase/.temp`

**Create:**
- `scripts/supabase-reset.sh` — runs `supabase db reset` to wipe local DB, apply migrations, apply seed
